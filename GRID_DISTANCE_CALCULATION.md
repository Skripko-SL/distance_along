# Расчёт расстояний от объектов до опорной сетки точек

Расчёт по дорожному графу — скрипт `distance_along_roads.py`.

---

## Исходные данные

### 1. Целевые объекты (school.dbf / .shp)

Файл в формате DBF (dBase/FoxPro), может сопровождаться shape-файлом.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | N | Уникальный идентификатор объекта |
| `xcoord` | N | X-координата (EPSG:3857, метры) |
| `ycoord` | N | Y-координата (EPSG:3857, метры) |

Другие поля (название, адрес и т.п.) передаются в выходной CSV как есть.

Проекция исходных данных — **Web Mercator (EPSG:3857)**, координаты в метрах (совпадают с геометрией .shp).

### 2. Опорная сетка (all_points.dbf)

Полная сетка точек переменного шага в проекции Web Mercator (EPSG:3857).
Содержит 707 489 точек разного шага (200×200, 400×400, 1000×1000 м).

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | N | Уникальный идентификатор точки сетки |
| `xcoord` | N | X-координата точки (EPSG:3857, метры) |
| `ycoord` | N | Y-координата точки (EPSG:3857, метры) |
| `col_index` | N | Индекс колонки в сетке |
| `row_index` | N | Индекс строки в сетке |



## Расчёт расстояний по дорожному графу (distance_along_roads.py)

Этот подход находит кратчайший путь от каждой точки сетки до ближайшей школы
по дорожной сети. Полное расстояние складывается из трёх компонент:

```
total_distance = перпендикуляр(школа → дорога)
               + расстояние_по_графу(проекция_школы → проекция_точки)
               + перпендикуляр(точка_сетки → дорога)
```

### Исходные данные

Дополнительно к school.dbf и all_points.dbf требуется:

| Файл | Описание |
|------|----------|
| `a_graf.shp` / `.dbf` / `.shx` | Дорожная сеть Нижегородской области (OSM, EPSG:3857), 164 112 сегментов |

Поля дорожной сети:

| Поле | Описание |
|------|----------|
| `fclass` | Класс дороги (primary, secondary, tertiary, residential, service, track, footway, path, etc.) |
| `oneway` | Направление: 'B' — двусторонняя, 'F' — односторонняя вперёд, 'T' — односторонняя назад |
| `leght` | Длина сегмента в метрах (предвычислена) |
| `name` | Название улицы |

### Метод

#### Этап 0. Перепроецирование в UTM 38N

Исходные данные (`school.dbf`, `all_points.dbf`, `a_graf.shp`) хранятся в **EPSG:3857 (Web Mercator)**. Эта проекция сохраняет углы, но искажает расстояния в `1/cos(широта)` раз — на широте Нижегородской области (~54.5–58°с.ш.) это даёт искажение **×1.72–1.89**. Если считать евклидово расстояние прямо по координатам EPSG:3857, все итоговые дистанции окажутся завышены примерно в эти же 1.7–1.9 раза (подробности — в разделе «Исправленные баги» ниже).

Поэтому перед построением графа и любыми геометрическими расчётами координаты объектов, сетки и дорог перепроецируются в **UTM 38N (EPSG:32638)** через `pyproj.Transformer` — эта зона покрывает Нижегородскую область почти без искажений (1 единица UTM = 1 истинный метр). Долгота/широта (WGS84) для вывода в CSV по-прежнему считаются напрямую из исходных EPSG:3857 координат (эта формула корректна и не требует UTM).

#### Этап 1. Построение дорожного графа и индекса сегментов

1. **Все вершины** каждой полилинии (в UTM 38N) становятся узлами графа (не только концы). Координаты округляются до 0.1 м для дедупликации — так соединяются сегменты, проходящие через одну точку, даже если их вершины различаются на сантиметры.
2. Ребро создаётся между каждой парой последовательных вершин в полилинии. Вес = евклидово расстояние между вершинами (уже в истинных метрах UTM).
3. Строится пространственный индекс (KD-дерево) из **всех вершин полилиний** (1 585 812 точек). Каждая вершина хранит ссылку на свой сегмент.

**Направленность рёбер (oneway) зависит от имени файла дорожной сети** (`build_road_graph` в `distance_along_roads.py` определяет это по basename без учёта регистра):

| Файл дорог | Граф | Учитывается ли `oneway` | Поведение |
|---|---|---|---|
| `a_graf.shp` | **Автомобильный** | Да | `oneway='F'` → ребро только по порядку вершин полилинии (prev → curr); `oneway='T'` → только в обратную сторону (curr → prev); `'B'`/иное/отсутствует → двустороннее ребро |
| `roads.shp` (или любое другое имя) | **Пеший** | Нет | Все рёбра двусторонние, как раньше — `oneway`, даже если он есть в DBF, полностью игнорируется (пешеход идёт по односторонней улице в обе стороны) |

Смысл разделения: скрипт используется для расчёта расстояний по двум разным графам — пешеходному (`roads.shp`, направление движения не ограничено) и автомобильному (`a_graf.shp`, движение ограничено `oneway`). Выбор графа — через `--roads <файл>`, никакого отдельного флага режима нет: имя файла однозначно определяет и граф, и то, нужно ли учитывать `oneway`.

Технически это реализовано так: рёбра графа хранятся в направлении, реально разрешённом движением. Чтобы получить расстояние «от точки сетки к школе» (а не наоборот) при односторонних дорогах, супер-источник + мульти-источниковый Dijkstra (Этап 3) запускается на **транспонированном** графе с `directed=True` — стандартный приём для расчёта кратчайших путей *к* фиксированному узлу в ориентированном графе. Для `roads.shp` рёбра и так симметричны, поэтому транспонирование ничего не меняет и поведение идентично прежнему неориентированному расчёту.

**Ключевое преимущество:** Из 1 347 331 уникальных узлов 206 260 (15.3%) являются общими для нескольких дорог (пересечения). Среди узлов-концов полилиний (250 839 шт.) общими являются 194 057 (77.4%) — то есть подавляющее большинство концов дорог физически стыкуются друг с другом, но без учёта промежуточных вершин граф на одних концах остаётся сильно фрагментированным (см. «Ключевое улучшение» ниже).

Характеристики графа (актуальный `a_graf.shp`, 164 112 сегментов):
- Узлов: **1 347 331** (все вершины полилиний, дедуплицированы)
- Рёбер: **2 843 394** (каждая пара последовательных вершин — двунаправленное ребро)
- Всего вершин полилиний (с дублями): **1 585 812**
- Компонент связности: **1** (граф полностью связный, 100% узлов в главной компоненте)

#### Этап 2. Проекция школ на дорожный граф (перпендикуляр)

Для каждой школы:
1. Через KD-дерево находятся K=3 ближайших вершины дорог
2. Для каждой уникальной дороги-кандидата вычисляется перпендикуляр от школы до полилинии
3. Выбирается дорога с **минимальным перпендикуляром** → `perp_school`
4. Определяется **доля** (`frac_s`) — положение основания перпендикуляра вдоль полилинии (0.0 = начало, 1.0 = конец)
5. Определяется **какой под-сегмент** полилинии содержит основание перпендикуляра: находятся два последовательных узла графа `(n1, n2)`, между которыми лежит перпендикуляр
6. Вычисляются расстояния от основания до каждого узла:
   - `dist_to_n1 = frac_s * total_len` (вдоль полилинии от начала до перпендикуляра)
   - `dist_to_n2 = (1 - frac_s) * total_len` (вдоль полилинии от перпендикуляра до конца)
7. В словарь `school_node_min` записывается:
   - `school_node_min[n1] = min(..., perp_school + dist_to_n1)`
   - `school_node_min[n2] = min(..., perp_school + dist_to_n2)`

#### Этап 3. Супер-источник и мульти-источниковый Dijkstra

Создаётся **виртуальный супер-узел**, соединённый рёбрами со всеми узлами, к которым есть доступ от школ. Вес ребра = `school_dist[node]`.

```
супер-узел ──[school_dist]──→ концевой_узел_A
          ──[school_dist]──→ концевой_узел_B
          ...
```

Запускается алгоритм Дейкстры от супер-узла. Результат — для каждого узла графа минимальное расстояние до ближайшей школы (уже включая перпендикуляр школы и смещение вдоль сегмента).

#### Этап 4. Проекция точек сетки на дорожный граф (перпендикуляр)

Для каждой точки сетки выполняется та же процедура, что и для школ (шаг 2):
1. Поиск ближайшей дороги (KD-дерево, K=3 кандидата)
2. Вычисление перпендикуляра → `perp_grid`
3. Определение доли вдоль полилинии → `frac_g`
4. Находятся два узла графа `(n1, n2)`, между которыми лежит основание перпендикуляра
5. Расстояние от точки до узлов: `d_n1` и `d_n2` (вдоль полилинии от перпендикуляра до каждого узла)

#### Этап 5. Формирование итогового расстояния

Для каждой точки сетки:

```
road_dist = min(
    d_n1 + node_dist[n1],   # d_n1 = dist вдоль полилинии от перпендикуляра до n1
    d_n2 + node_dist[n2]    # d_n2 = dist вдоль полилинии от перпендикуляра до n2
)
total = perp_grid + road_dist
```

где `node_dist[n]` — уже включает `perp_school` + смещение школы вдоль её сегмента (результат Дейкстры от супер-источника).

#### Примечание о связности графа

Благодаря использованию **всех вершин полилиний** как узлов графа (с округлением до 0.1 м), граф дорожной сети на актуальном `a_graf.shp` стал **полностью связным** (1 компонента, 100% узлов). Все 707 489 точек сетки достижимы по дорогам.

#### Исправленные баги

При переходе на текущий формат данных (`xcoord`/`ycoord` в метрах EPSG:3857 вместо смеси градусов и границ ячеек) и последующей эксплуатации были обнаружены и исправлены три независимых бага, задвоивших итоговые расстояния:

1. **(09.07.2026) Градусы вместо метров.** После смены формата атрибутов код по инерции продолжал ждать старые поля/конвертацию градусы→меркатор для координат объектов, из-за чего расстояния оказались задвоены.
2. **(09.07.2026) Неверный брекетинг в `get_foot_bracket_vertices`.** Расстояние от основания перпендикуляра до узлов-скобок считалось от начала/конца **всей полилинии дороги**, а не от самих узлов сегмента (нужно было `target - prev_cum`, а не просто `target`, аналогично для второго узла), плюс off-by-one в условии остановки цикла. Это систематически завышало `road_dist` для дорог с 3+ вершинами (у используемой сети в среднем ~9.7 вершины на полилинию — то есть почти всех дорог).
3. **(10.07.2026) Расчёт расстояний в Web Mercator вместо равнопромежуточной проекции.** Все геометрические расчёты (`euclidean_dist` для перпендикуляров и рёбер графа) велись прямо по координатам EPSG:3857. Web Mercator — конформная проекция: она сохраняет углы, но искажает масштаб по формуле `1/cos(широта)`. На широте Нижегородской области (~54.5–58°с.ш.) это давало систематическое завышение всех расстояний в **1.72–1.89 раза** («почти в два раза»). Исправлено перепроецированием всех координат в UTM 38N (EPSG:32638) перед расчётом расстояний (см. «Этап 0» выше). Проверено сверкой с независимым пешеходным/автомобильным маршрутизатором (OSRM, `routing.openstreetmap.de`) на нескольких контрольных парах школа↔точка: расхождение после фикса — доли процента (например, 6 400 м расчёт против ~6 373 м по факту), тогда как до фикса расчёт был почти вдвое больше реального маршрута.

Проверяется неравенством треугольника: `total_dist_m` не может быть меньше прямого (евклидова, в UTM) расстояния от точки сетки до её `source_id`. Актуально — 0 нарушений на выборке 20 000 точек, медианный коэффициент извилистости `total_dist_m / прямое_расстояние` ≈ 1.37, что типично для дорожной сети.

**Важно (осталось не исправлено):** дорожная сеть `a_graf.shp` почти не содержит чисто пешеходной инфраструктуры (footway/path/pedestrian — суммарно 44 км из 87 290 км сети, 0.05%). Для маршрутов внутри населённых пунктов, где в реальности пешеход срезает через двор или парк, расчёт может быть заметно (на 10–50%) больше реального пешеходного маршрута — это ограничение исходных данных, а не баг кода.

**Важно:** `id` в `all_points.dbf` не уникален (707 489 записей, но только 702 735 уникальных `id`). Для сверки/джойна результата по строкам использовать позицию строки (порядок совпадает с порядком в DBF) или пару `col_index`+`row_index`, а не один `id`.

## Использование для других объектов

### Подготовка данных

Для расчёта расстояний от **любых других объектов** (больницы, остановки, магазины, etc.) необходимо подготовить DBF-файл с полями:

```csv
id, xcoord, ycoord
1, 4787053.03, 7416953.45
2, 4820361.04, 7435601.03
```

Минимально обязательные поля: `id`, `xcoord`, `ycoord` (EPSG:3857, метры).
Опционально: `id_t` — строковый идентификатор.

### Запуск расчёта

```bash
python3 distance_along_roads.py --objects hospitals.dbf
```

Результат: `all_points_to_hospitals_distance.csv`.

### Форматы входных данных

Скрипт принимает:

| Формат | Библиотека | Пример |
|--------|-----------|--------|
| DBF (dBase/FoxPro) | `dbfread` | `DBF(path, raw=True)` |
| CSV | встроенный `csv` | `csv.DictReader(open(path))` |

### Требования к системе координат

| Данные | Ожидаемая проекция |
|--------|--------------------|
| Целевые объекты | **EPSG:3857** (Web Mercator), координаты `(xcoord, ycoord)` в метрах |
| Опорная сетка | **EPSG:3857** (Web Mercator), координаты в метрах |

Если ваши данные в другой проекции — перед расчётом необходимо выполнить перепроецирование (например, через `pyproj.Transformer`).

### Зависимости

```bash
pip3 install scipy numpy dbfread pyshp pyproj
```

Установлено в системе.
